## 1. Report Generation Overview

### 1.1 Report Types

PolicyCheck AU delivers **two report variants** based on engagement stage:

| Report Type | When Delivered | Length | Content | Disclaimer |
|-------------|---|---|---|---|
| **Teaser Report** | In cold outreach email; free | 1-2 pages | APP compliance score, top 3 gaps, "See full report" CTA | Standard disclaimer + "abbreviated" notice |
| **Full Report** | After lead conversion; paid | 5-8 pages | All 13 APPs scored, gap analysis, recommendations, sector context, audit trail | Full legal disclaimer |

## 2. Report Content Standards

### 2.1 Quality Checks Before Generation

**Automated Pre-Generation Validation:**

| Check | Failure Action |
|-------|---|
| Analysis confidence ≥ threshold for tier | Stop; request analyst review |
| All 13 APPs scored (or documented as "N/A" with reason) | Stop; run Mode B again |
| No contradictions in APP scores (e.g., APP 6 score 1.0 but gap says "no disclosure") | Stop; analyst flag |
| Recommendations are specific, not generic | Fail; analyst rewrites |
| Analyst sign-off present | Stop; route to analyst |
| Report ID unique (not duplicate) | Stop; generate new ID |

## 5. Report Quality Checks (Pre-Delivery)

**Mandatory 3-step QA before report sent to prospect:**

### 5.1 Automated QA (System)
```
✓ Report ID unique
✓ Disclaimer language unmodified (exact match)
✓ All 13 APPs scored or marked "N/A"
✓ Scores internally consistent (no contradictions)
✓ Recommendations specific (not generic)
✓ No profanity or offensive language
✓ Links functional (if report is interactive)
✓ Analyst signature present
✓ Date & time accurate
✓ Prospect name/company correct
```

### 5.2 Analyst QA (Human Review)
```
✓ Policy excerpts accurately represent policy (sample-check 3 excerpts)
✓ Recommendations are proportionate to gap severity
✓ Sector context appropriate & accurate
✓ Tone is professional, non-judgmental
✓ Confidence score reflects analysis quality
✓ No legal conclusions ("you are in violation")
```

### 5.3 Delivery QA (Final Gate)
```
✓ Recipient email verified in CRM
✓ Tier confirmation matched to order
✓ Delivery method correct (email/portal/both)
✓ Branding consistent with tier
✓ No personally identifiable information in filenames
✓ PDF secure (encrypted if Premium)
```

**If Any Check Fails:** Report not sent; returned to analyst for correction.

## 7. Report Versioning & Updates

### 7.1 Versioning Rules

**Reports are static documents (not auto-updated) because:**
- Client paid for analysis as-of a specific date
- Retroactively changing reports creates liability
- Regulatory changes require new analysis, new report

**Versioning Scheme:**
```
Report ID: PC-[YYYYMMDD]-[SEQUENTIAL]-v[VERSION]
Example:   PC-20260401-12345-v1.0

Version changes:
v1.0 → Original report
v1.1 → Minor correction (typo, link fix)
v1.2 → Analyst-requested clarification
v2.0 → Reanalysis under new APP definitions (rare)
```

### 7.2 Update Triggers & Protocol

**When analysis should be re-run (new report):**
1. **Client-initiated:** Client updated policy; requests new assessment
2. **Regulatory Change:** APP definitions/guidance materially changed; old report marked "Legacy"
3. **Dispute:** Client disputes findings; analyst re-reviews & creates v2.0

**When correction required (v1.1):**
1. Factual error in report (e.g., policy quote inaccurate)
2. Typo or formatting issue
3. Broken link

**Update Notification:**
- Teaser: No notification (free product)
- Standard: Email client with v1.1 update link
- Premium: Auto-update in portal + email notification
